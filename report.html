CityNexus Smart City OS

Process and Thread Management Framework for Modern Smart Cities

Project: CEP-3
Course: Operating Systems
Date: December 2025

1. Executive Summary

CityNexus is an innovative Smart City Operating System that applies core OS principles to urban management. By treating city services as processes and their sub-tasks as threads, we create an efficient, scalable, and intelligent framework for managing complex urban infrastructure. This project demonstrates the practical application of process scheduling, thread management, memory allocation, and inter-process communication in a real-world context.

Key Innovation: Bridging traditional OS concepts with modern smart city requirements, enabling real-time resource allocation, priority-based emergency response, and seamless inter-service coordination.

2. Problem Statement

Modern cities face increasing complexity in managing multiple interconnected services simultaneously. Traditional city management systems lack:

  • Efficient resource allocation across competing services
  • Real-time priority adjustment for emergencies
  • Systematic inter-service communication mechanisms
  • Scalable architecture for adding new services
  • Performance monitoring and optimization capabilities

This project addresses these challenges by designing an OS-level framework that treats urban services as computational processes, enabling efficient scheduling, resource management, and coordination.

3. System Architecture

3.1 Layered Architecture

CityNexus follows a five-layer architecture model:

  1. Application Layer: City services (Traffic, Emergency, Utilities, Transport, Security)
  2. Process Management Layer: Process scheduler, priority manager, resource allocator
  3. Thread Management Layer: Thread scheduler, synchronization primitives, deadlock prevention
  4. Resource Management Layer: Memory manager, I/O scheduler, disk scheduler
  5. Hardware Layer: CPU cores, memory, storage, network devices, IoT sensors

3.2 Service as Process Model

Each city service is modeled as an independent process with:

  • Unique Process ID (PID)
  • Priority level (0 = highest for emergencies, 3 = lowest)
  • Memory allocation requirement
  • Multiple threads for sub-tasks
  • State transitions (New → Ready → Running → Waiting → Terminated)

4. Process Management

4.1 Scheduling Algorithm

CityNexus implements a hybrid scheduling algorithm combining:

  • Priority-based Preemptive Scheduling: Emergency services can preempt lower-priority processes
  • Round Robin: Equal time quantum for same-priority processes to prevent starvation
  • Aging mechanism: Gradually increases priority of waiting processes

4.2 Process State Diagram

4.3 Context Switching

Context switches occur when:

  • A higher-priority process arrives (preemption)
  • Time quantum expires for current process
  • Process enters waiting state (I/O operation)
  • Process terminates

The system tracks context switches for performance analysis and optimization.

5. Thread Management

5.1 Multi-threading Model

Each service process contains multiple threads executing concurrently:

  • Traffic System: Signal Control, Vehicle Detection, Route Optimization
  • Emergency Service: 911 Dispatcher, Ambulance Routing, Hospital Coordination
  • Security System: Camera Monitor, Threat Detection, Alert System

5.2 Synchronization Mechanisms

To prevent race conditions and ensure data consistency:

  • Mutex Locks: Protect shared resources (traffic signals, databases)
  • Semaphores: Control access to limited resources (emergency vehicles)
  • Message Queues: Inter-thread communication

5.3 Deadlock Prevention

Implementation of Banker's Algorithm to ensure system never enters deadlock state:

  • Resource allocation graphs maintained
  • Safe state verification before allocation
  • Circular wait prevention through resource ordering

6. Memory Management

6.1 Memory Allocation Strategy

CityNexus implements dynamic memory allocation with:

  • Total System Memory: 1024 MB
  • Allocation Policy: First-fit algorithm
  • Protection: Each process has isolated memory space
  • Monitoring: Real-time tracking of memory usage

6.2 Virtual Memory

For large-scale data handling:

  • Paging mechanism for efficient memory utilization
  • LRU (Least Recently Used) page replacement
  • Demand paging for city databases and sensor data

6.3 Memory Management Analysis

CLO-3 Fulfillment: Memory allocation strategies compared:

  • First-fit: Fast allocation, moderate fragmentation
  • Best-fit: Minimal fragmentation, slower allocation
  • Worst-fit: Large holes remain, useful for future allocations

Chosen Strategy: First-fit for real-time performance requirements

7. Disk Scheduling Algorithms

7.1 Application in CityNexus

City services require frequent database access for:

  • Traffic patterns and historical data
  • Emergency response records
  • Utility consumption logs
  • Security camera footage
  • Public transport schedules

7.2 Algorithm Comparison

Algorithm Seek Time Starvation Fairness Best Use Case
FCFS High No Excellent Low load scenarios
SSTF Low Yes Poor Performance-critical
SCAN Medium No Good Balanced workload
C-SCAN Medium No Excellent Heavy load (chosen)

Disk Scheduling Visualization - C-SCAN Algorithm

7.3 Implementation Choice

Selected Algorithm: C-SCAN (Circular SCAN)

Rationale: Provides uniform wait time for all services, prevents starvation, and performs efficiently under high load - critical for smart city operations with continuous database access from multiple services.

8. Inter-Process Communication (IPC)

8.1 Communication Mechanisms

City services need to coordinate and share information:

  • Message Passing: Emergency alerts to all services
  • Shared Memory: Real-time traffic data shared with transport system
  • Pipes: Log data from services to monitoring system
  • Sockets: Network communication with external systems

8.2 Example Scenario

Emergency Response Coordination:

  1. Emergency service receives 911 call (highest priority)
  2. Sends message to Traffic system to clear route
  3. Traffic system updates signal timings via shared memory
  4. Security system receives notification via message queue
  5. All systems coordinate through IPC mechanisms

9. Performance Metrics & Analysis

9.1 Key Performance Indicators

  • CPU Utilization: Percentage of time CPU is active
  • Throughput: Number of processes completed per time unit
  • Response Time: Time from request to first response (critical for emergencies)
  • Turnaround Time: Total time from submission to completion
  • Context Switches: Number of process switches (overhead indicator)
  • Memory Utilization: Percentage of memory in use

9.2 Trade-off Analysis

Requirement Solution Trade-off
Fast emergency response Priority-based preemption May delay low-priority services
Fair resource distribution Round-robin for same priority Slightly higher context switching overhead
Prevent starvation Aging mechanism Additional computational overhead
Maximize throughput SSTF disk scheduling Potential starvation (rejected)

12. Challenges Faced & Solutions

12.1 Challenge: Priority Inversion

Problem: Low-priority utility service holding resources needed by emergency service.

Solution: Priority inheritance protocol - temporarily elevate priority of resource-holding process.

12.2 Challenge: Starvation of Low-Priority Services

Problem: Continuous emergency requests could starve utility management.

Solution: Aging mechanism gradually increases priority of waiting processes.

12.3 Challenge: Memory Fragmentation

Problem: Frequent process creation/termination causes memory fragmentation.

Solution: Compaction algorithm runs periodically during low-load periods.
