⚙️ Our CPU Scheduling Algorithm

Priority-Based Preemptive Scheduling

Algorithm: Multi-Level Priority Queue Scheduling

Algorithm Steps:

1. When process arrives → Check priority level (0-3)
2. IF priority == 0 (Emergency) THEN
    Preempt current process immediately
    Allocate CPU to emergency process
3. ELSE
    Add to appropriate priority queue
4. Schedule from highest priority non-empty queue
5. Use Round Robin within same priority level

📊 Key Features

  • Preemptive scheduling
  • Priority aging to prevent starvation
  • Dynamic priority adjustment
  • Context switching optimized

⏱️ Time Complexity

  • Process insertion: O(1)
  • Next process selection: O(1)
  • Priority check: O(1)
  • Queue switching: O(1)

📊 Algorithm Comparison

Why Priority Scheduling Over Others?

❌ FCFS (First Come First Serve)

Problem: If ambulance arrives after 10 street light processes, it waits until all finish. Result: Lives lost due to delay.

⚠️ Round Robin

Problem: Ambulance gets same time slice as street light. If time quantum = 2s, ambulance must wait for other processes' turns. Result: Unpredictable emergency response.

⚠️ Shortest Job First (SJF)

Problem: Long emergency processes (like coordinating multiple ambulances) wait for shorter tasks. Result: Complex emergencies delayed.

✅ Priority-Based Scheduling (OUR CHOICE)

Advantage: Emergency processes (Priority 0) ALWAYS execute first, regardless of arrival time or duration. Result: Guaranteed instant response for critical services!

🔄 What is a Process?

Understanding City Service Processes

Definition

A process is a running instance of a city service program. Each process has its own memory space, process ID (PID), priority level, and execution state.

🚑 Example: Ambulance Process

  • PID: AMB_001
  • Priority: 0 (Emergency)
  • State: Running
  • Memory: GPS data, route map, patient info
  • Threads: GPS tracking, siren control, route planning

💡 Example: Street Light Process

  • PID: LIGHT_245
  • Priority: 3 (Normal)
  • State: Ready/Waiting
  • Memory: Timing schedule, sensor data
  • Threads: Timer, brightness sensor
Key Point: Each city service runs as an independent process with its own resources and priority level!

🧵 What are Threads?

Lightweight Tasks Within Processes

Definition

A thread is a lightweight sub-task within a process. Multiple threads can run concurrently, sharing the same memory space but executing different tasks simultaneously.

🚑 AMBULANCE PROCESS (Priority 0)
↓ Splits into 3 concurrent threads ↓
📍 Thread 1
GPS Tracking
(Updates location every 1s)
🔊 Thread 2
Siren Control
(Manages sound patterns)
🗺️ Thread 3
Route Planning
(Calculates fastest path)
⚡ Advantage: All 3 threads run simultaneously - ambulance tracks location, sounds siren, and plans route AT THE SAME TIME for faster response!

🔄 Process vs Thread

Understanding the Difference

🔷 PROCESS

  • Independent execution unit
  • Own memory space
  • Own process ID (PID)
  • Heavy context switching
  • Inter-process communication needed
  • Example: Entire ambulance service
VS

🧵 THREAD

  • Lightweight sub-task
  • Shared memory with parent
  • Own thread ID (TID)
  • Fast context switching
  • Direct memory sharing
  • Example: GPS tracking within ambulance
💡 Analogy: Think of a process as a restaurant (ambulance service) and threads as individual workers (GPS, siren, route) - they all work in the same kitchen (shared memory) but do different tasks!

📞 System Calls

Interface Between User and Kernel

What are System Calls?

System calls are the programmatic way for processes to request services from the operating system kernel. In CityNEXUS, they're used to create emergency processes, allocate resources, and manage city services.

1️⃣ Citizen dials 911 emergency
↓
2️⃣ System Call: create_process("ambulance")
↓
3️⃣ Kernel validates request
↓
4️⃣ Set Priority = 0 (Emergency)
↓
5️⃣ Allocate CPU & Memory resources
↓
6️⃣ Create threads (GPS, Siren, Route)
↓
7️⃣ Execute immediately - Ambulance dispatched!

🔄 Context Switching

How CPU Switches Between Processes

What is Context Switching?

When CPU switches from one process to another, it saves the current process state and loads the new process state. This is called context switching.

Street Light Process Running
↓
🚨 AMBULANCE ARRIVES (Priority 0)
↓
Save Street Light State (registers, PC, stack)
↓
Load Ambulance State
↓
Ambulance Executes
↓ After completion ↓
Restore Street Light & Resume
⚡ Speed: Context switch takes only 1-2 microseconds in CityNEXUS!

🔒 Deadlock Prevention

Ensuring System Never Gets Stuck

What is Deadlock?

When two or more processes wait for each other to release resources, creating a circular dependency. System gets stuck!

❌ Deadlock Example

Ambulance A has GPS resource, needs Siren

Ambulance B has Siren, needs GPS

Both wait forever = DEADLOCK!

✅ Our Solution

  • Resource ordering protocol
  • Timeout mechanisms
  • Priority inheritance
  • Resource preemption for emergencies
Guarantee: Emergency processes can preempt any resource - NO deadlock for Priority 0!

⚙️ Process Synchronization

Coordinating Concurrent Execution

Why Synchronization?

When multiple processes access shared resources, we need synchronization to avoid race conditions and data corruption.

🔐 Mutex (Mutual Exclusion)

Only one process can access resource at a time

Use: Traffic signal controller - only one direction gets green

🚦 Semaphores

Controls access to limited resources

Use: Parking slots - only N ambulances in garage

Emergency Override: Priority 0 processes can break synchronization locks in emergencies!

⏱️ Real-Time Operating System Features

Hard vs Soft Real-Time

🔴 Hard Real-Time (Emergency)

  • Deadline MUST be met
  • Missing deadline = catastrophic failure
  • Examples: Ambulance dispatch, Fire alert
  • Guaranteed response: < 100ms

🔵 Soft Real-Time (Normal)
